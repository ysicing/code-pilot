# 任务：[FEATURE NAME]

**输入**：来自 `.claude/specs/[###-feature-name]/` 的设计文档
**前置条件**：plan.md（必需）、spec.md（用户故事需要）、research.md、data-model.md、contracts/

**测试**：下面的示例包括测试任务。测试是可选的 - 仅在功能规范中明确请求时才包括它们。

**组织**：任务按用户故事分组，以支持每个故事的独立实现和测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行运行（不同文件，无依赖）
- **[Story]**：此任务属于哪个用户故事（例如，US1、US2、US3）
- 在描述中包含准确的文件路径

## 路径约定

- **单个项目**：存储库根目录下的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **Mobile**：`api/src/`、`ios/src/` 或 `android/src/`
- 下面显示的路径假设单个项目 - 根据 plan.md 结构调整

## 依赖项与执行顺序

### 阶段依赖项

- **设置（第 1 阶段）**：无依赖 - 可立即开始
- **基础（第 2 阶段）**：取决于设置完成 - 阻挡所有用户故事
- **用户故事（第 3+ 阶段）**：所有都取决于基础阶段完成
  - 用户故事可以并行进行（如果有人员安排）
  - 或按优先级顺序进行（P1 → P2 → P3）
- **打磨（最后阶段）**：取决于所有必需用户故事完成

### 用户故事依赖项

- **用户故事 1 (P1)**：可在基础（第 2 阶段）完成后开始 - 与其他故事无依赖
- **用户故事 2 (P2)**：可在基础（第 2 阶段）完成后开始 - 可与 US1 集成，但应独立可测试
- **用户故事 3 (P3)**：可在基础（第 2 阶段）完成后开始 - 可与 US1/US2 集成，但应独立可测试

### 在每个用户故事内

- 必须先编写测试（如果包含）并在实现前失败
- 模型优于服务
- 服务优于端点
- 核心实现优于集成
- 故事完成后再转向下一个优先级

### 并行机会

- 所有标记为 [P] 的设置任务可并行运行
- 所有标记为 [P] 的基础任务可并行运行（在第 2 阶段内）
- 基础阶段完成后，所有用户故事可并行开始（如果团队能力允许）
- 故事中所有标记为 [P] 的测试可并行运行
- 故事中标记为 [P] 的模型可并行运行
- 不同用户故事可由不同团队成员并行处理

## 实现策略

### MVP 优先（仅用户故事 1）

1. 完成第 1 阶段：设置
2. 完成第 2 阶段：基础（关键 - 阻挡所有故事）
3. 完成第 3 阶段：用户故事 1
4. **停止并验证**：独立测试用户故事 1
5. 部署/演示（如果就绪）

### 增量交付

1. 完成设置 + 基础 → 基础已就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事增加价值而不破坏以前的故事

### 并行团队策略

有多个开发者时：

1. 团队一起完成设置 + 基础
2. 基础完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 故事完成并独立集成

## 说明

- [P] 任务 = 不同文件、无依赖
- [Story] 标签将任务映射到特定用户故事以进行追踪
- 每个用户故事应该可以独立完成和测试
- 验证测试在实现前失败
- 在每个任务或逻辑组后提交
- 在任何检查点停止以独立验证故事
- 避免：模糊任务、相同文件冲突、破坏独立性的故事间依赖
