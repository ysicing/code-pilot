## 用户输入

```text
$ARGUMENTS
```

在继续之前，您**必须**考虑用户输入（如果不为空）。

## 概要

用户在触发消息中 `/spec-kit:specify` 后输入的文本**就是**功能描述。即使 `$ARGUMENTS` 在下方按字面显示，也要假设您在此对话中始终可获得它。除非用户提供空命令，否则不要要求他们重复。

给定该功能描述，请执行以下操作：

1. **生成简洁的短名称**（2-4 个字）用于分支：
   - 分析功能描述并提取最有意义的关键字
   - 创建一个 2-4 个字的短名称，捕捉功能的本质
   - 尽可能使用动作-名词格式（例如"add-user-auth"、"fix-payment-bug"）
   - 保留技术术语和缩写（OAuth2、API、JWT 等）
   - 保持简洁但具有描述性，能够一目了然理解功能
   - 示例：
     - "I want to add user authentication" → "user-auth"
     - "Implement OAuth2 integration for the API" → "oauth2-api-integration"
     - "Create a dashboard for analytics" → "analytics-dashboard"
     - "Fix payment processing timeout bug" → "fix-payment-timeout"

2. **在创建新分支之前检查现有分支**：

   a. 首先，获取所有远程分支以确保我们拥有最新信息：

      ```bash
      git fetch --all --prune
      ```

   b. 为短名称查找所有源中最高的 feature 编号：
      - 远程分支：`git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
      - 本地分支：`git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
      - Specs 目录：检查匹配 `specs/[0-9]+-<short-name>` 的目录

   c. 确定下一个可用编号：
      - 从所有三个源中提取所有编号
      - 找到最高编号 N
      - 对新分支编号使用 N+1

   d. 使用计算的编号和短名称运行脚本 `$HOME/.claude/scripts/specify/create-new-feature.sh --json "$ARGUMENTS"`：
      - 传递 `--number N+1` 和 `--short-name "your-short-name"` 以及功能描述
      - Bash 示例：`$HOME/.claude/scripts/specify/create-new-feature.sh --json "$ARGUMENTS" --json --number 5 --short-name "user-auth" "Add user authentication"`
      - PowerShell 示例：`$HOME/.claude/scripts/specify/create-new-feature.sh --json "$ARGUMENTS" -Json -Number 5 -ShortName "user-auth" "Add user authentication"`

   **重要**：
   - 检查所有三个源（远程分支、本地分支、specs 目录）以查找最高编号
   - 仅匹配具有确切短名称模式的分支/目录
   - 如果未找到该短名称的现有分支/目录，从编号 1 开始
   - 每个功能只能运行此脚本一次
   - JSON 在终端中作为输出提供 - 始终参考它以获取您要查找的实际内容
   - JSON 输出将包含 BRANCH_NAME 和 SPEC_FILE 路径
   - 对于参数中的单引号，例如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）

3. 加载 `$HOME/.claude/templates/specify/spec-template.md` 以了解必需的部分。

4. 遵循此执行流程：

    1. 从输入解析用户描述
       如果为空：ERROR "未提供功能描述"
    2. 从描述提取关键概念
       识别：参与者、操作、数据、约束
    3. 对于不清楚的方面：
       - 基于背景和行业标准进行知情的猜测
       - 仅在以下情况下标记 [NEEDS CLARIFICATION: 具体问题]：
         - 选择显著影响功能范围或用户体验
         - 存在具有不同含义的多个合理解释
         - 不存在合理的默认值
       - **限制：最多 3 个 [NEEDS CLARIFICATION] 标记**
       - 按影响优先处理澄清：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填充用户场景和测试部分
       如果没有清楚的用户流程：ERROR "无法确定用户场景"
    5. 生成功能需求
       每个需求必须可测试
       对未指定的细节使用合理的默认值（在假设部分中记录假设）
    6. 定义成功标准
       创建可测量、技术无关的结果
       包括定量指标（时间、性能、体积）和定性措施（用户满意度、任务完成）
       每个标准必须无需实现细节即可验证
    7. 识别关键实体（如果涉及数据）
    8. 返回：SUCCESS（规范准备好规划）

5. 使用模板结构将规范写入 SPEC_FILE，用从功能描述（参数）派生的具体细节替换占位符，同时保留部分顺序和标题。

6. **规范质量验证**：写入初始规范后，根据质量标准进行验证：

   a. **创建规范质量检查清单**：使用检查清单模板结构在 `FEATURE_DIR/checklists/requirements.md` 生成检查清单文件，包含这些验证项：

      ```markdown
      # 规范质量检查清单：[FEATURE NAME]

      **目的**：在进行规划之前验证规范的完整性和质量
      **创建日期**：[DATE]
      **功能**：[链接到 spec.md]

      ## 内容质量

      - [ ] 无实现细节（语言、框架、API）
      - [ ] 专注于用户价值和业务需求
      - [ ] 为非技术利益相关者编写
      - [ ] 所有强制部分已完成

      ## 需求完整性

      - [ ] 没有 [NEEDS CLARIFICATION] 标记保留
      - [ ] 需求可测试且明确
      - [ ] 成功标准可测量
      - [ ] 成功标准是技术无关的（无实现细节）
      - [ ] 所有验收场景已定义
      - [ ] 边界情况已识别
      - [ ] 范围清晰界定
      - [ ] 依赖关系和假设已识别

      ## 功能就绪性

      - [ ] 所有功能需求都有明确的验收标准
      - [ ] 用户场景涵盖主要流程
      - [ ] 功能满足成功标准中定义的可测量结果
      - [ ] 没有实现细节泄入规范

      ## 注意事项

      - 标记为不完整的项目需要在 `/spec-kit:clarify` 或 `/spec-kit:plan` 之前更新规范
      ```

   b. **运行验证检查**：根据每个检查清单项审查规范：
      - 对于每项，确定它是否通过或失败
      - 记录发现的具体问题（引用相关规范部分）

   c. **处理验证结果**：

      - **如果所有项都通过**：标记检查清单完成并继续第 6 步

      - **如果项失败（排除 [NEEDS CLARIFICATION]）**：
        1. 列出失败的项和具体问题
        2. 更新规范以解决每个问题
        3. 重新运行验证直到所有项通过（最多 3 次迭代）
        4. 如果 3 次迭代后仍然失败，在检查清单注释中记录剩余问题并警告用户

      - **如果 [NEEDS CLARIFICATION] 标记保留**：
        1. 从规范中提取所有 [NEEDS CLARIFICATION: ...] 标记
        2. **限制检查**：如果存在超过 3 个标记，仅保留 3 个最关键的（按范围/安全性/UX 影响）并对其余部分做出知情的猜测
        3. 对于每个需要澄清的部分（最多 3 个），以此格式向用户展示选项：

           ```markdown
           ## 问题 [N]：[主题]

           **背景**：[引用相关规范部分]

           **我们需要了解的内容**：[NEEDS CLARIFICATION 标记中的具体问题]

           **建议的答案**：

           | 选项 | 答案 | 含义 |
           |--------|--------|--------------|
           | A      | [第一个建议答案] | [这对功能意味着什么] |
           | B      | [第二个建议答案] | [这对功能意味着什么] |
           | C      | [第三个建议答案] | [这对功能意味着什么] |
           | 自定义 | 提供您自己的答案 | [说明如何提供自定义输入] |

           **您的选择**：_[等待用户响应]_
           ```

        4. **关键 - 表格格式**：确保 markdown 表格格式正确：
           - 使用一致的间距，管道对齐
           - 每个单元格应在内容周围有空格：`| Content |` 而不是 `|Content|`
           - 标题分隔符必须至少有 3 个短划线：`|--------|`
           - 测试表格在 markdown 预览中正确呈现
        5. 按顺序对问题进行编号（Q1、Q2、Q3 - 最多 3 个）
        6. 在等待响应之前一起显示所有问题
        7. 等待用户为所有问题提供选择（例如"Q1: A, Q2: Custom - [details], Q3: B"）
        8. 通过用用户的选定或提供的答案替换每个 [NEEDS CLARIFICATION] 标记来更新规范
        9. 在所有澄清解决后重新运行验证

   d. **更新检查清单**：在每次验证迭代后，用当前的通过/失败状态更新检查清单文件

7. 使用分支名称、规范文件路径、检查清单结果和下一阶段的就绪性报告完成（`/spec-kit:clarify` 或 `/spec-kit:plan`）。

**注意**：该脚本在写入之前创建并检出新分支并初始化规范文件。

## 一般指南

## 快速指南

- 专注于**用户需要什么**和**为什么**。
- 避免**如何**实现（无技术栈、API、代码结构）。
- 为业务利益相关者而非开发人员编写。
- 不要在规范中创建嵌入的检查清单。这将是一个单独的命令。

### 部分需求

- **强制部分**：必须为每个功能完成
- **可选部分**：仅当与功能相关时才包含
- 当部分不适用时，完全删除它（不要留作"N/A"）

### 对于 AI 生成

从用户提示生成此规范时：

1. **做知情的猜测**：使用背景、行业标准和常见模式来填补空白
2. **记录假设**：在假设部分中记录合理的默认值
3. **限制澄清**：最多 3 个 [NEEDS CLARIFICATION] 标记 - 仅用于关键决策：
   - 显著影响功能范围或用户体验
   - 具有不同含义的多个合理解释
   - 缺乏任何合理的默认值
4. **优先处理澄清**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **像测试人员一样思考**：每个模糊需求应该失败"可测试且明确"检查清单项
6. **需要澄清的常见区域**（仅当不存在合理的默认值时）：
   - 功能范围和边界（包含/排除特定用例）
   - 用户类型和权限（如果存在多个冲突的解释）
   - 安全/合规需求（当法律/财务重大时）

**合理的默认示例**（不要询问这些）：

- 数据保留：该域的行业标准实践
- 性能目标：标准的 Web/Mobile 应用期望，除非指定
- 错误处理：用户友好的消息，具有适当的备选方案
- 认证方法：Web 应用的标准基于会话或 OAuth2
- 集成模式：RESTful API，除非另有指定

### 成功标准指南

成功标准必须是：

1. **可测量的**：包括具体指标（时间、百分比、计数、速率）
2. **技术无关的**：不提及框架、语言、数据库或工具
3. **以用户为中心**：从用户/业务角度描述结果，而不是系统内部
4. **可验证的**：无需知道实现细节即可测试/验证

**好的示例**：

- "用户可以在 3 分钟内完成结账"
- "系统支持 10,000 并发用户"
- "95% 的搜索在 1 秒内返回结果"
- "任务完成率提高 40%"

**不好的示例**（以实现为中心）：

- "API 响应时间在 200 毫秒以下"（太技术性，使用"用户立即看到结果"）
- "数据库可以处理 1000 TPS"（实现细节，使用面向用户的指标）
- "React 组件高效呈现"（特定于框架）
- "Redis 缓存命中率超过 80%"（特定于技术）
